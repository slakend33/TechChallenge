# Tech Challenge вҖ” Terraform

Production-grade Azure infrastructure deployed via Terraform. Provisions a multi-tier environment with virtual networking, compute VMs, an internal load balancer, Key Vault, and Storage вҖ” all wired together through a strict module dependency chain with no hard-coded values.

## Module Structure

```
.
в”ңв”Җв”Җ main.tf                  # Root orchestration вҖ” module calls and dependency chain
в”ңв”Җв”Җ variables.tf             # Input variable declarations
в”ңв”Җв”Җ outputs.tf               # Root-level outputs
в”ңв”Җв”Җ terraform.tfvars         # Environment-specific values
в””в”Җв”Җ modules/
    в”ңв”Җв”Җ network/             # VNet, subnets, service endpoints
    в”ңв”Җв”Җ compute/             # VMs, NICs, availability set, SSH keys
    в”ңв”Җв”Җ storage/             # Storage account with subnet firewall
    в”ңв”Җв”Җ security/            # Key Vault, Service Principal, RBAC
    в”ңв”Җв”Җ loadbalancer/        # Internal LB, backend pool, health probe
    в””в”Җв”Җ nsg/                 # Network Security Groups for web and mgmt subnets
```

## Deployment Stages

The root `main.tf` enforces a strict ordering using `depends_on` and `time_sleep` resources to account for Azure control-plane propagation delays.

| Stage | Resources | Delay |
|-------|-----------|-------|
| **0** | TLS key pairs (local, no Azure API) | вҖ” |
| **1** | `module.network` + `module.storage` (parallel) | 30s after network |
| **2** | `module.compute` + `module.security` (parallel) | 60s after compute |
| **3** | `module.loadbalancer` | 30s after LB |
| **4** | `module.nsg` | вҖ” |

---

## Key Design Decisions

**SSH keys are Terraform-managed.** RSA 4096-bit key pairs are generated by the `tls` provider at apply time. Public keys are injected into VM resources; private keys are stored as Key Vault secrets (`ssh-private-key-web`, `ssh-private-key-mgmt`). No keys are written to local disk.

**Storage is subnet-locked.** The Storage Account firewall permits traffic only from the management subnet via a `Microsoft.Storage` service endpoint вҖ” no public internet access.

**No hard-coded resource IDs.** All cross-module references (NIC IDs, subnet IDs, private IPs, storage account ID) are passed through module outputs, keeping the configuration fully dynamic.

**Remote state is opt-in.** The `azurerm` backend block is commented out in `main.tf`. Uncomment and run `terraform init -migrate-state` after the first apply once the Storage Account exists.

## Quick Start

```bash
# 1. Authenticate
az login
az account set --subscription "<your-subscription-id>"

# 2. Update terraform.tfvars with your subscription_id, tenant_id,
#    a globally unique storage_account_name and key_vault_name,
#    and your public IP in kv_ip_rules.

# 3. Init, plan, apply
terraform init
terraform fmt -recursive && terraform validate
terraform plan -out=tfplan
terraform apply tfplan
```

Full step-by-step instructions are in [`Terraform_VS_Code_Deployment_Guide.docx`].

## Retrieving SSH Keys

Private keys are never written locally. Retrieve them from Key Vault after apply:

```bash
az keyvault secret show \
  --vault-name cfc-prod-001 \
  --name ssh-private-key-web \
  --query value -o tsv > ssh-web.pem && chmod 600 ssh-web.pem

az keyvault secret show \
  --vault-name cfc-prod-001 \
  --name ssh-private-key-mgmt \
  --query value -o tsv > ssh-mgmt.pem && chmod 600 ssh-mgmt.pem
```
